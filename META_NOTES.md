## 1. 哪些 patch 方向在 Epoch 1-3 有效

### 1.1 「不能 X」改写为「机制 + 对策 + 典型对话」三段式 — ✅ 有效

- **作用**：把 `06-failure-diagnosis-playbook.md` / `04-risk-tags.md` / `11-exception-handling.md` / `12-risk-escalation.md` 的所有"不能"段落补齐机制解释 + 可执行对策。
- **为什么有效**：原 17 条黑名单中至少 5 条 + 4 个故障 + 10 个风险 + 12 类异常的所有"不能承诺"都缺机制说明——这是 **SkiillLen D1 失败机制编码** 的最大缺口。
- **写作模式**：每条故障/风险/异常都按以下结构组织：
  - 现状（典型表现 / 触发信号）
  - 机制（为什么不能直接归因 / 承诺）
  - 可执行对策（必查的 P0/P1 字段 + 必看的证据 + 必走的检查顺序）
  - 典型对话（✗ 错误回应 + ✓ 正确回应）
- **验证**：抽 3 个故障场景，看修复后是否能从"业务员单点说法"走到"必查 P0/P1 清单"。

### 1.2 黑名单祈使句改写为「绝不要 X，因为 Y，如果必须做则 Z」三段式 — ✅ 有效

- **作用**：把 `SKILL.md` 末尾 17 条 Forbidden Behavior 沉淀到 `references/31-blacklist-with-causality.md`。
- **为什么有效**：原 17 条全部是祈使句，**没有 "because Y" 因果链**——这是 **SkiillLen D3 高风险动作黑名单** 的最大缺口。
- **写作模式**：每条黑名单按 3 列组织：
  - 绝不要做（X）—— 具体动作，不抽象
  - 机制 / 损失（Y）—— 损失类型 + 影响范围
  - 替代动作（Z）—— 2-3 条具体替代路径
- **验证**：grep `references/31-blacklist-with-causality.md` 确认 17 行（#1 到 #17）每行三列非空。

### 1.3 在 `SKILL.md` 顶部 L02 后插入 `<!-- SLOW_UPDATE_START -->` 受保护区段 — ✅ 有效

- **作用**：把跨轮次战略约束（失败优先 / 黑名单硬约束 / 承诺边界 / 可视化 hand-off / 引用一致性）沉淀到受保护区段。
- **为什么有效**：Step-level 编辑器不能改受保护区段 = 战略约束不被 patch 漂移。
- **写作模式**：直接、祈使、含触发条件。
- **验证**：`grep -c "<!-- SLOW_UPDATE_START -->" SKILL.md` 应为 1。

### 1.4 「建议证据」后追加「如何拿到」小段 — ✅ 轻度有效

- **作用**：D2 加固，让业务员/客户知道**怎么拿到**这些数据（不仅知道要什么，还知道怎么要）。
- **为什么有限**：4 个文件已经按 Epoch 1 的"可执行对策"补了字段名，"如何拿到"是锦上添花。
- **建议**：下一轮优化**优先度低**——除非有具体证据表明业务员不知道如何提供某项数据。

### 1.5 `references/32-self-check-rubric.md` 输出前自检 — ✅ 有效（结构性补丁）

- **作用**：把 SkiillLen 的"输出前自检"通用三问与 oneclear 业务特定 6 项合并。
- **为什么有效**：把 4 轮 patch 沉淀的"硬约束"集中到一个可被训练模型 Step-level 调用的清单——**patch 之后**每轮输出前都过一遍，防止退化。
- **写作模式**：Q1-Q3 通用 + Q4-Q9 oneclear 特定 + Q10-Q12 跨问题组合。
- **建议**：未来 patch 之后，**强制跑一次 Q1-Q12** 作为回归。

### 1.6 Convergence 末尾"保存本地 Markdown + AskUserQuestion 询问(不提 skill 名字) + 缺装跳出来再问" — ✅ 有效（解耦可视化决策与安装决策）

- **作用**：把"要不要可视化"和"是否安装"两件事彻底解耦——先无条件保存本地 Markdown（用户始终拿到交付物），再用 AskUserQuestion 询问（**用户视角不刻意出现 skill 名字**），缺装时**跳出来再用 AskUserQuestion 询问**（不默认跳过、不自动装）。
- **为什么有效**：
  - 原"自动调用"流程在用户没看到 Markdown 时就触发可视化；
  - 原"缺装三选一"把"可视化决策"和"安装决策"耦合，且文案直接出现 skill 名字。
  - 重构后两阶段完全解耦，**用户始终拿到本地 Markdown**（可视化是可选增强），**用户视角只看到"是否需要制作展示素材"**（不提 skill），**缺装时仍由用户决定**（不剥夺决策权）。
- **写作模式**：
  1. 无条件保存到 `~/.cc-switch/oneclear-outputs/convergence-<timestamp>-<case-id>.md`
  2. AskUserQuestion 单题 2 选项（chip `可视化`，文案"需要制作"/"不需要"）
  3. 用户选"需要制作" → 探测工具 → INSTALLED 调用 / MISSING 跳出来再问
  4. MISSING AskUserQuestion 单题 2 选项（chip `安装`，文案"立即安装"/"稍后手动安装"）
  5. 用户选"不需要" → 结束本轮 /oneclear 调用
- **验证**：跑 6 个 self-check 验证用例时，第 6 个（Convergence 可视化）应走新流程，且 2b 选项文案不含 skill 名字。

### 1.7 决策点文件 D1 化（08 / 13 三段式） — ✅ 有效（覆盖高频使用场景）

- **作用**：把 08 业务员模糊表达（5 类）和 13 合规边界（7 类）从"可能对应/Agent处理"改写为"**机制（为什么）+ 步骤化对策（怎么走）+ 允许/禁止话术 + 升级条件**"四段式。
- **为什么有效**：这两份文件是售前最常被引用的——08 出现在每轮 Round 1 提取工況时，13 出现在每次触达合规边界时。**把这两份 D1 化 = 把高频使用场景的"机制 + 对策"密度提到与 04/06 同级**。
- **写作模式**：
  - 08：每条模糊表达 = 机制（为什么不能字面理解）+ 对策（拆几个维度）+ 典型对话（✗ 错误 + ✓ 正确）
  - 13：每类合规 = 机制（为什么会触发）+ 步骤化对策（怎么走流程）+ 允许/禁止话术 + 升级条件
- **建议**：未来 D1 化新文件时，**先看该文件是否高频引用**——高频的优先（08/13/19/20/23），低频的可以延后。

### 1.8 决策点文件 D3 化（19 / 20 / 23 因果黑名单化） — ✅ 有效（让禁止承诺可执行）

- **作用**：把 19 试用 / 20 竞品 / 23 Level 4-5 的"禁止承诺 / 禁止话术"段从祈使句改写为"**绝不要做 X，因为 Y**"三段式因果黑名单。
- **为什么有效**：原"禁止承诺"段只说"不要 X"——没说为什么，导致训练模型 / 业务员在压力下容易绕过。**改成"因为 Y"后，每条禁止都有清晰的损失类型 + 影响范围，违反前会先想到后果**。
- **写作模式**：
  - 表头：`绝不要做 | 机制（为什么） | 替代动作/话术`
  - 每条因果：必含具体损失类型（法律责任 / 商业损失 / 客户信任崩塌）+ 影响范围（Agent / 销售 / 客户）
- **建议**：未来"禁止承诺"段改写时，**先看是否高频 + 是否已触达 13-compliance-and-commitment.md**——高频且触达合规的优先。

### 1.9 5 步 Convergence D2 加固（2a 命令级 + 2c-yes 错误处理 + 09 如何填） — ✅ 有效（让 D2 强项更密）

- **作用**：把 Convergence 末尾 5 步流程从"描述"补到"可执行"：
  - 2a 加具体 mkdir/写文件 shell 命令 + 时间戳 + case-id 派生
  - 2c-yes 加错误处理（超时 / 工具返回错误 / 用户主动取消 / Refinement 二次触发）
  - 09-output-templates 加"如何填"工具/命令级指引（8 字段逐一对应）
- **为什么有效**：原 2a 只说"保存到本地 Markdown"——没说怎么保存。**补具体命令 = agent 不再需要自行推断目录、文件名、写入方式**。原 2c-yes 没说错误处理——超时/失败时 agent 可能陷入死等。**补错误处理 = 边界明确，agent 知道该退出 / 提示 / 重试**。
- **建议**：未来 D2 加固时，**先看该段是否在"高频被调用 + 原描述抽象"**——抽象的优先（2a/2c-yes/09 都属于这类）。

## 2. 哪些 patch 方向在 Epoch 1-3 有害 / 应当避免

### 2.1 改写 7 个 Python 脚本的内部逻辑 — ❌ 有害

- **原因**：脚本是 **guardrail**（护栏），不是 patch 目标。改逻辑 = 引入新 bug 风险。
- **例外**：仅在 `validate_output_pack.py` 报错信息不清晰时可补错误码文档；**不改逻辑**。

### 2.2 重写 `30-askuserquestion-integration.md` / `27-interactive-round-protocol.md` — ❌ 有害

- **原因**：这两份已是 oneclear 的 D2 强项，**重写会破坏 D2**。Epoch 1-3 没碰它们，验证后 D2 强项保持。
- **例外**：除非发现 30 中某 chip 命名与 AskUserQuestion 工具约束冲突（≤12 字符）；**优先改 chip 名，不改协议**。

### 2.3 在受保护区段里塞业务规则 — ❌ 有害

- **原因**：受保护区段是 **战略约束**（跨轮次），不是业务规则。塞业务规则 = 战略与战术混淆。
- **正确做法**：业务规则放具体 reference 文件；战略约束放受保护区段。

### 2.4 在示例里重写所有 8 个 — ❌ 有害（over-engineering）

- **原因**：示例是 patch 的"演示"，不是 patch 的"主战场"。全写 8 个 = 大量重复。
- **正确做法**：下一轮只挑 2-3 个代表性示例（`existing-project-failure.md` / `quote-before-discovery.md` / `tender-tech-bid.md`）按新结构改写。

### 2.5 把"低优先 patch"误判为"高优先" — ❌ 有害（rank 失真）

- **原因**：ranking 的 4 维（systematic impact × complementarity × generality × actionability）必须**全部评估**，不能只凭 1 维拍板。
- **典型错误**：D2 加固（如何拿到）的 actionability 高，但 systematic impact 低 — 优先级应低于 D1 / D3。

## 3. 写更好的 patch 应该遵循的原则

### 3.1 失败优先 + 成功补充（merge_final 规则）

- D1 失败 patch（修复系统性误判） > D3 黑名单 patch（修"为什么不能"） > D2 加固 patch（保守补充）
- 永远不要因为 D2 写得好就把 D1 退后。

### 3.2 通用原则 > 任务专有

- 写"任何故障/风险开口前必查 3 项中的至少 2 项" > 写"糊袋要查压差曲线"
- 写"任何 framing trap 必须走反问证据" > 写"上家掺旧料要反问检测报告"

### 3.3 已存在内容不重复

- 写 patch 前先 grep 看目标文件是否已覆盖——避免在两处分别维护导致漂移。
- 31-blacklist 与 SKILL.md Forbidden Behavior 段保持引用关系，**不在两处分别维护 17 条**。

### 3.4 优先级 = 修复系统性反复出现 > 修一个边缘 case

- 修一个能解决 50% 失败模式的 patch > 修 10 个边缘 case 的 patch。
- 评估 ranking 时**优先看 systematic impact**。

### 3.5 受保护区段 = 只由 slow_update 写

- Step-level 编辑器**不**应尝试改 `<!-- SLOW_UPDATE_START -->` 内。
- 改受保护区段的触发条件：epoch boundary / 跨多轮反复出现的元规则 / 评估回归失败。

### 3.6 自检清单 = 每轮输出前过

- Q1-Q3 通用三问（必答）
- Q4-Q9 oneclear 特定项（按当前轮阶段选答）
- Q10-Q12 跨问题组合（仅 epoch 边界 / 重大 patch 后跑）

## 4. 失败模式库（避免重复踩坑）

### 4.1 "不能 X" 段只补了「机制」、没补「对策」

- **症状**：Q1 自检 ✗（只说"为什么不能"，没说"怎么才能"）
- **修复**：把抽象动词（"建议核查/合理评估"）替换为具体动作（"调 `assess_context_level()` / 在工況档案 `[未知]` 字段写 P0/P1 名称"）

### 4.2 黑名单合并了多条 / 漏了某些原 17 条

- **症状**：Q11 自检 ✗（行数 ≠ 17）
- **修复**：grep 31-blacklist 与原 SKILL.md 17 条对比，找缺漏。

### 4.3 受保护区段被 Step-level patch 改写

- **症状**：Q12 自检 ✗（受保护区段内容漂移）
- **修复**：回滚到上一次 epoch 的受保护区段 + 评估为什么 Step-level 编辑器越界。

### 4.4 在 SKILL.md 与 31-blacklist 两处分别维护 17 条

- **症状**：Q12 自检 ✗（grep 发现两处有不同步的内容）
- **修复**：删除 SKILL.md 末尾的 17 条（已替换为引用），只留 31-blacklist 单一来源。

### 4.5 D1 修复破坏了 D2 强项

- **症状**：Q10 自检 ✗（D1 修后 D2 抽象）
- **修复**：补"可执行对策"段时，**必须含具体工具名 + 字段名**——不允许"建议核查"等抽象动词。

## 7. 本文件使用方式（给未来优化者）

1. 启动下一轮（epoch N+1）时，**先读本文件 §1-4**——了解哪些方向有效、哪些有害、哪些模式要避免。
2. 写 patch 草稿时，**对照本文件 §3.1-3.6**——确保 patch 符合"失败优先 / 通用优先 / 不重复 / 系统性优先 / 受保护区段只读 / 自检通过"6 原则。
3. patch 完成后，**跑 §6 的 6 项一致性检查**——任一 ✗ = 漂移。
4. epoch 结束前，**更新本文件 §1-5**——加入新发现的"有效方向"和"有害方向"；保留旧条目（除非被证伪）。
5. epoch 结束前，**更新本文件 §5（未来方向）**——给下一轮优化者留具体建议。

本文件不应被训练模型直接读取（meta 层，不是业务规则）。
