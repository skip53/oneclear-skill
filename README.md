# OneClear

> 把销售转述、客户原话、现场工况和报价压力，整理成一份可推进、可追问、可控风险的工业除尘滤料售前判断。

OneClear 是一个面向工业除尘滤料售前技术支持的 Agent Skill，可在 Claude Code、Codex、OpenCode、Cursor 等 agent 中运行。

---

## 解决什么问题

工业除尘滤料的售前场景里，技术支持每天面对的输入大多是这样的：

- 客户只说一句话——"压差高""寿命短""竞品太贵""能不能先报个价"。
- 销售只转述半句——把客户原话压缩成"客户怀疑是材料质量问题"。
- 关键工况（温度、含湿、粉尘特性、清灰方式、当前材质、故障证据）往往一个都没问。

这时候有三个常见的坑：

1. **把客户的表层诉求当成真实问题**：客户说"便宜点、寿命差不多就行"，不代表真的可以降配；客户说"是不是材料质量问题"，不代表技术支持要直接认领或否认。
2. **信息不足就出技术判断**：在没有压差曲线、袋面照片、工况数据的情况下，直接给材质方向、报价或寿命承诺，等于把责任和风险一起背了。
3. **没有节奏地一次性问一堆问题**，或者反过来什么都不问就憋出一份"看起来完整"的诊断报告。

OneClear 要顶替的，正是技术支持人工做的这第一遍工作：**听转述 → 挑出真实工况问题 → 判断信息够不够 → 决定现在能说什么、必须先问什么**。它不是通用销售话术工具，也不会替你做最终选型签字——它的价值是把这段"听-问-判断"的过程结构化、分级、留痕，让销售知道下一句该问什么，让技术判断不会在证据不够时被推得太远。

---

## 怎么工作

OneClear 不是一次性甩给你一份报告，而是**一轮一轮跟你来回确认**：

```text
第 0 轮   先确认要做什么          → 给一个任务菜单，明确这次要处理的是哪类事
第 1 轮   把工况整理成档案        → 挑出最关键的信息，判断是哪类问题，标出风险信号
第 2 轮   一项一项往下问          → 每轮问 3-7 个问题，并告诉你现在进度到哪了
  ...                              每轮都会重新判断"现在能说到什么程度"
第 N 轮   信息攒够了，给出结论    → 输出完整的诊断材料
后续补充  有新情况随时更新结论    → 不用整套流程重新走一遍
卡住了    遇到红线就停下来        → 把风险讲清楚，给几条替代路径让你选
```

每个关键节点（确认任务类型、确认是哪类问题、要不要现在就下结论、要不要顺便做一份可视化展示、卡住时怎么办）都会跳出一个可以直接点选的选择框，不需要你打字回复数字来"模拟选择题"。

能说什么、不能说什么由一个 **0-5 级的"判断深度"** 决定——级别越低，能说的越保守，只给追问和澄清；只有信息攒够了，级别才会往上走，才允许给方向、给候选方案。产品里这个级别叫 Level，详见下方[核心机制](#核心机制)。

> 关键不变量：OneClear **永远先问任务菜单**，再做任何分析。即使用户一次粘贴了完整对话，也会先确认任务类型，再抽取事实、列缺失字段。

## 安装

```bash
npx skills add skip53/oneclear-skill
```

这条命令会通过 Skills CLI 自动装到你正在用的 agent（Claude Code / Codex / OpenCode / Cursor）。安装后用 `$oneclear` 调用。

如果 Skills CLI 不可用，可以用 npm fallback：

```bash
npx oneclear install
```

默认会装到所有支持的 agent；加 `--target codex` / `claude` / `opencode` / `cursor` 可只装其中一个。

## 完整示范：从一句话转述到正式输出包

下面是一个连贯的端到端走读，展示 OneClear 真实的多轮交互长什么样。场景：水泥窑尾袋式除尘器用了半年压差升高、清灰下不来，销售怀疑是滤料质量问题，想直接换方案。

### 第 0 轮：先确认要做什么

```text
业务员: $oneclear
客户说袋子用了半年压差就很高，清灰也没啥用，问是不是袋子质量不行，
销售想直接给客户换个方案。
```

OneClear 不会立刻分析，而是先给出任务菜单 + 已识别到的事实，并给出推荐默认：

```markdown
你今天需要处理哪个任务?（选一个最贴的,后面可以再调）

A) 新项目选型   B) 旧项目故障   C) 替换/降本   D) 报价准备
E) 试用验证     F) 招投标支持   G) 竞品对比    H) 复购/补单
I) 我有客户原话/聊天记录   J) 其他/不确定

已识别到的事实:
- [已知] 故障表现: 压差高, 清灰后不下降
- [已知] 使用时长: 约半年
- [未知] 当前滤料材质、工况温度、含湿、粉尘特性、清灰参数

推荐默认: B 旧项目故障（销售提到"换方案"，但故障原因还未确认）
```

> 这里会跳出一个选择框，问"按这个默认任务来，还是要调整"：「确认默认: B 旧项目故障」/「调整为: C 替换·降本」/「调整为: 其他」/「其他(可粘原文)」——业务员直接点选 **B**，不用打字回复。

### 第 1 轮：把工况整理成档案

业务员选完任务后，OneClear 会先确认一次"这到底是哪类问题"，再正式开始整理：

> 又跳出一个选择框：「新项目选型」/「旧项目故障」/「替换·降本」/「信息不足」——业务员点选 **旧项目故障**。

OneClear 输出 8 块工况档案草稿、初步风险标签，以及当前能确定到什么程度（Level）：

```markdown
## 第 1 轮 / 预计 4 轮 · 当前 Level 1

- 任务类型: 旧项目故障   路径: 旧项目故障
- 案件状态: intake

### 已知信息 (✅)
- [已知] 故障表现: 压差高,清灰后不下降
- [已知] 使用时长: 约半年

### 未知关键信息 (❌)
- [未知] 当前滤料材质/后处理 — 为什么问: 判断是否真的是材质问题的前提
- [未知] 压差曲线 — 为什么问: 区分逐步糊袋/突发堵塞/清灰异常
- [未知] 工况温度、含湿、粉尘粘性 — 为什么问: 判断糊袋/结露风险
- [未知] 袋面照片或拆袋样品 — 为什么问: 故障定位需要证据,不能只靠转述

### 初步风险标签
- 糊袋风险: 压差高 + 清灰失效信号,缺含湿/粘性证据

### 本案承诺红线（持续更新）
- 不承诺换材质一定解决
- 不承认或否认质量问题
- 不直接推荐替换方案
```

> 这一轮结尾会一次性问完几件事：这轮里哪些问题业务员现在就能答上来（比如"压差曲线""当前材质"打个勾），接下来由谁去补剩下的信息——业务员选了**自己去找资料补**。

### 第 2-3 轮：一项一项把信息补完

业务员陆续补上压差曲线、当前材质（涤纤针刺毡）、工况温度（约 180℃ 偶尔上探）、袋面照片。OneClear 每轮把新信息并进档案、重新判断现在能说到什么程度、更新进度：

```markdown
## 第 2 轮 / 预计 4 轮 · 当前 Level 2

- 本轮新增: 当前材质=涤纤针刺毡; 温度约180℃偶尔上探; 提供压差曲线
- 上一轮结论: 故障表现明确,材质/工况待补

### 矛盾信息 (⚠️)
- [矛盾] 客户说"稳定运行多年" vs 销售转述"半年就糊袋" — 影响: 时间线不一致,先要求客户确认实际更换/启用时间,不能据此下结论
```

```markdown
## 第 3 轮 / 预计 1 轮 · 当前 Level 3

- 本轮新增: 袋面照片显示局部发硬结块; 含湿确认偏高
- 已知 11/12，剩余 1 项：清灰压力/周期参数

### 初步风险标签
- 糊袋风险: 含湿偏高 + 局部温度上探，疑似间歇性结露
```

### 信息攒够了：给出正式结论

判断深度够了之后，OneClear 不会自动把结论甩给你，而是先确认一次：

> 跳出一个选择框：「现在就出结论」/「继续追问」/「我还有补充」——业务员选了**现在就出结论**。

随后输出正式的诊断材料，分四块（当前 Level 3：能给选型方向，但还不能定具体型号）：

```markdown
## 信息够了，给出结论 · 当前 Level 3

### 工况档案（完整版）
...（8 块工况档案，已知/未知/矛盾全部标注）

### 追问清单
内部技术版 + 客户转述版，列出仍待确认的清灰参数

### 初步诊断摘要
更像间歇性结露引发的局部糊袋，不是单纯材质质量问题；当前涤纤针刺毡在偏高含湿+温度波动工况下耐受边界值得关注，但不构成"材质有问题"的结论。

### 禁止承诺项（本案专列，收尾红线）
- 不承诺换材质一定解决
- 不承认或否认原材质质量问题
- 不在缺清灰参数前给出最终选型方向
```

OneClear 先把这份结果**保存成一份本地 Markdown 文件**（`./oneclear-output/convergence-<时间戳>.md`），再问一句：

> 「要不要顺便做一份可视化展示素材？」——「需要制作」/「不需要」——业务员选了**需要制作**。

如果检测到可视化工具（`huashu-design`）已经装好，OneClear 会把保存好的文件和这份诊断结论交给它去做成可视化 HTML（信息图/长文/幻灯片三选一）；如果没装，会再问一句"现在装"还是"先不装、自己手动装"——不会自作主张帮你装上，也不会假装没看到就直接跳过。

## 能力一览

| 能力 | 输出 | 典型用途 |
|---|---|---|
| **任务菜单识别** | A-J 任务清单 + 推荐默认 | 确认业务员要做什么，再做任何分析 |
| **多轮推进** | 第 0 轮 → 第 N 轮 → 给出结论 | 任务分阶段、按轮收集信息、避免一次性输入 |
| **进度条显示** | 第 X 轮 / 预计 N 轮 · Level L | 让业务员实时知道自己在哪、还差什么 |
| 工况档案整理 | 八块工况档案草稿 | 把零散客户原话、销售转述、已知/未知/矛盾信息结构化 |
| 案件分流 | A-H 任务类型 + Level 0-5 评估 | 避免还没搞清楚问题就进入报价或选型 |
| 真实需求判断 | 表层诉求 vs 真实工况问题 vs 商业目标 | 识别"想便宜一点"背后的稳定性、寿命、排放或停机风险 |
| 风险标签 | 温度、含湿、腐蚀、磨蚀、糊袋、火星、清灰、压差等 | 提醒销售不要遗漏关键风险 |
| 追问清单 | 3-7 个最优先要确认的问题，含内部技术版和客户转述版 | 让销售知道下一轮该问什么、怎么问 |
| 报价准备度 | 不可报价 / 预算报价 / 人工确认后正式报价候选 | 防止"先报个价"变成技术承诺 |
| Level 4/5 售前包 | 候选方向、证据要求、试用边界、禁止承诺、人工确认 | 支撑竞品替换、试用验证、招投标、复杂项目推进 |

## 核心机制

1. **先判断真实问题，再谈方案** — 区分客户原话、销售转述、已知事实和矛盾点，不把"想便宜一点""怀疑材质问题"这类表层诉求直接当成真实需求。
2. **先确认任务，再启动诊断** — 永远先弹任务菜单确认要处理的是哪类事，再开始分析，避免报价和选型需求被混在一起处理。
3. **按轮次推进，封顶 7 轮** — 每轮问 3-7 个问题、显示进度，最多 7 轮必须给出结论，不会没完没了地问下去。
4. **用 0-5 级（Level）卡住能说的话** — 级别越低越保守，只能问澄清；级别够了才允许给方向、给候选方案：

   | Level | 能说到什么程度 |
   |---|---|
   | 0 | 只能追问，不能分流 |
   | 1 | 能分流，不能诊断 |
   | 2 | 能初步诊断，不能选型 |
   | 3 | 能给方向，不能定型号 |
   | 4 | 能给候选方案，需人工确认 |
   | 5 | 能出完整技术支持包 |

5. **脚本是护栏，不是裁判** — `scripts/` 里的确定性检查器只负责分类、缺失字段、风险标签和规则库校验，不输出最终型号/寿命/排放承诺；和参考文档冲突时取更严格的边界。
6. **黑名单必须讲因果，不能静默绕开** — `references/31-blacklist-with-causality.md` 里的 17 条禁止动作都标注了机制/损失和替代方案，违反时必须显式承认理由，不能悄悄绕过。
7. **可视化是解耦的可选项** — 结论先无条件存一份本地 Markdown，再问是否需要可视化；要做且工具已装好才会调用，没装会问是否安装，不会自作主张帮你装上或直接跳过。

## 局限性

- OneClear 不替代最终专家签字、现场检查、实验室检测、合同技术承诺或质保承诺。
- 信息不足时，它会降级为追问和风险提示，不会强行推荐具体材料型号。
- 对排放、寿命、替换效果、竞品优劣、招投标响应等内容，会保留人工确认和禁止承诺项。
- npm fallback 的 Cursor 安装是项目级安装；如果希望跨项目复用，优先使用 `npx skills add ...` 的全局安装能力。
