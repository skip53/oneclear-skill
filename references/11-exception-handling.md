# 11｜异常处理协议

异常不是失败，而是 Agent OS 必须显式接管的状态。

## 异常类型

| 异常 | 触发信号 | 处理方式 |
| --- | --- | --- |
| 客户原话缺失 | 只有业务员概括 | 要求补原话，降低承诺等级 |
| 无法分流 | 看不出新项目、故障或替换 | 只问最小澄清问题 |
| 表层需求与工况冲突 | 想降本但现场不稳定，想报价但无参数 | 先做真实需求验证 |
| 信息矛盾 | 一边说稳定，一边说压差高或寿命短 | 列为矛盾信息，不做结论 |
| 证据不足 | 缺照片、样品、压差曲线、检测报告 | 只输出证据需求 |
| 过度承诺请求 | 保证寿命、保证排放、完全替代 | 标注承诺风险，要求人工确认 |
| 责任归因压力 | 客户问是否质量问题或索赔 | 不承认不否认，先要求证据 |
| 全量替换压力 | 无试用边界就要求替代竞品 | 建议小批量试用或分仓验证 |
| 责任归因陷阱 | 客户或业务员直接断言"上家掺旧料/你们材质问题" | 不承认不否认,显式列出反问证据(同批是否同工况、是否有检测、是否有工况变化),把责任问题隔离到证据链后 |
| 信息矛盾 | 客户一边说稳定,一边说破袋或要求替换 | 拆分两面陈述,标为 contradictory_facts,只输出矛盾清单与追问,不推进方案 |
| 保密越界 | 客户带竞品出厂报告/检测数据/PPT 资料,或业务员顺手评价竞品 | 资料先归档脱敏,不评论竞品优劣,所有判断回到我方工况和基线 |
| 客户原话丢失 | 微信/会议纪要里业务员转述掺入自己的"是不是材质问题"等评价 | 强制分离 customer_original_words 与 salesperson_interpretation,缺原话时标 context_gap |

## 输出要求

异常状态下，必须输出：

- 异常类型。
- 为什么这是异常。
- 现在允许说什么。
- 现在禁止说什么。
- 下一步需要谁补什么证据。

## 升级规则

以下异常需要人工技术确认：

- 质保、寿命、排放、合同、招投标承诺。
- 客户索赔或质量责任判断。
- 大项目全量替换。
- 安环风险、火星烧袋、爆炸或环保处罚风险。
- 客户资料、竞品报告、检测数据涉及保密边界。
