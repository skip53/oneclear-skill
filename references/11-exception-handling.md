# 11｜异常处理协议

异常不是失败，而是 Agent OS 必须显式接管的状态。
本文件每条异常都按「触发信号 + 机制 + 步骤化对策 + 允许/禁止话术 + 升级条件」结构组织——读完一条，就知道**这种异常为什么会发生**、**按什么顺序处理**、**可以/不可以对客户说什么**、**什么情况下必须升级到人**。

> 通用原则：异常状态下**禁止走技术输出**——所有的承诺、推荐、归因都暂停，**只输出「现在在什么异常 + 现在允许说什么 + 缺什么证据 + 谁来补」**四件事。

---

## 客户原话缺失

- **触发信号**：只有业务员概括，没有客户原始话术（微信截图 / 邮件 / 会议纪要）。
- **机制（为什么会触发）**：业务员转述会**压缩、改写、添加评价**——"客户说压差高，是不是滤料不行？" 中「是不是滤料不行」是业务员加的评价，不是客户原话。**按业务员转述做技术判断 = 把"业务员判断"当"客户工况"**，技术方案与现场失真。
- **步骤化对策**：
  1. **暂停技术输出**——不归因、不推荐、不承诺。
  2. **明确要客户原话**——微信截图 / 邮件原文 / 会议纪要 ≥ 1 份，**不接受业务员口头转述**。
  3. **拆出原话与评价**——拿到原话后，强制分离 `customer_original_words` 与 `salesperson_interpretation` 两个字段。
  4. **降承诺等级**——原话未拿到前，所有输出按 Level ≤ 1 处理。
- **允许说什么**：「需要客户原话才能诊断。请发微信截图 / 邮件 / 会议纪要。」
- **禁止说什么**：「可能是…」「建议…」「大概因为…」（任何含"可能/大概/建议"的技术输出）。
- **升级条件**：连续 2 轮仍拿不到原话 → 转人工（销售负责人复核）。

## 无法分流

- **触发信号**：输入中看不出「新项目 / 故障 / 替换」任一类型——纯问候、纯问题、纯文件但无语义。
- **机制（为什么会触发）**：分流是所有后续判断（Level 评估、风险标签、追问清单、输出模板）的前置；**分流失败 = 后续判断无锚点**。强行往下走会进入"按最像的猜一个"的模式，**导致后续 3-7 轮全部建立在错误的分流上**。
- **步骤化对策**：
  1. **暂停技术输出**。
  2. **只问最小澄清问题**——A-J 任务菜单 10 个选项 + 1 个"其他（可粘原文）"。
  3. **不要猜**——不要按"最像新项目"猜一个然后往下走；猜了就退不出来。
- **允许说什么**：「你今天需要处理哪个任务？」 + 完整 A-J 菜单。
- **禁止说什么**：任何基于"猜测"分流的技术判断。
- **升级条件**：用户选 J（其他/不确定）且不补内容 → 停在 Round 0，等用户补。

## 表层需求与工况冲突

- **触发信号**：客户/业务员说「想降本」但现场「不稳定」（频繁破袋 / 压差高 / 排放超标）；或「想报价」但现场「无参数」。
- **机制（为什么会触发）**：表层需求 ≠ 真实工况问题 ≠ 商业目标。**表层需求是客户/业务员"想要的"，真实工况是"现场实际的"，商业目标是"客户最终要的"**。三者冲突时按表层需求推进 = 客户拿到不解决现场问题的方案 → 不稳定 + 频繁更换 → 比原方案更贵。
- **步骤化对策**：
  1. **暂停推荐**——不按"降本/报价"直接出方案。
  2. **做真实需求验证**——走 `references/10-true-demand-validation.md` 的 gap 分析（表层 / 工况 / 商业目标三层）。
  3. **列出 gap**——把"客户说想要的"与"现场实际需要的"显式拆开。
  4. **回到证据**——所有 gap 必须有现场数据/照片/曲线佐证，不靠业务员转述。
- **允许说什么**：「表层需求与现场工况有 gap，请先确认现场状态——是否有照片/曲线/数据？」
- **禁止说什么**：「既然想降本，可以推荐…」（绕过 gap 直接出方案）。
- **升级条件**：gap 不可调和（客户拒绝承认现场问题）+ 仍要求推进 → 标 red，转人工。

## 信息矛盾

- **触发信号**：客户一边说「稳定运行」，一边说「频繁破袋 / 压差高 / 要求全量替换」。
- **机制（为什么会触发）**：矛盾陈述往往**来自不同时间窗或不同口径**——"过去 12 个月稳定运行" + "最近 1 个月频繁破袋"可能不矛盾（工况漂移）；"稳定" + "频繁破袋"在同时间窗下**几乎必有一方失实**。**按任一方推进 = 拿矛盾当事实，方案必然错**。
- **步骤化对策**：
  1. **不归因**——明确标 `contradictory_facts`，不取均值、不选信一方。
  2. **拆时间窗 / 拆口径**——分别问"什么时候开始频繁破袋"+"稳定是哪段时间的稳定"，把矛盾拆为两个独立事实。
  3. **列矛盾清单**——把双方陈述并列展示，不评价、不解释。
  4. **不推进方案**——矛盾未解决前不进入 Level 4+ 输出。
- **允许说什么**：「陈述 X（稳定）与陈述 Y（频繁破袋）存在矛盾。请分别说明时间窗与口径。」
- **禁止说什么**：「既然稳定运行，可能是…」（任选一边推进）；「忽略破袋情况，先按稳定…」（绕过矛盾）。
- **升级条件**：用户坚持矛盾陈述不变 + 仍要求推进 → 标 red，转人工。

## 证据不足

- **触发信号**：缺照片、缺样品、缺压差曲线、缺检测报告、缺运行记录。
- **机制（为什么会触发）**：技术判断 = 证据 → 假设 → 推论。**证据缺失 = 推论无锚 = 推荐失真**。在滤料场景，**单凭"压差高"或"破袋"等单点症状推荐型号，准确率 < 30%**（见 06-failure-diagnosis-playbook.md 的多因机制）。
- **步骤化对策**：
  1. **不归因**——不基于缺失证据出"诊断"或"方案"。
  2. **列证据需求**——按 06 / 04 对应故障 / 风险给出**必查证据清单**（具体到 P0/P1 字段）。
  3. **标注证据缺口**——在工況档案里把所有 `[未知]` 字段显式列出。
  4. **保持 Round 状态**——证据未补前，Level 评估为 0–1，不进入 Converge。
- **允许说什么**：「要开口给方向，需要 ① … ② … ③ … 三项证据，凑齐至少 2 项。」
- **禁止说什么**：「没有证据也大概能判断是…」（基于缺失证据的判断）。
- **升级条件**：连续 2 轮仍拿不到任何证据 → 标 amber（关键未知），转人工。

## 过度承诺请求

- **触发信号**：客户/业务员要求「保证排放达标 / 保证寿命 2 年 / 完全替代竞品 / 一定能降本 X%」。
- **机制（为什么会触发）**：承诺 = 法律 / 商业责任。**Agent 无现场试用的承诺 = 拿公司信誉对赌**——一旦工况与承诺条件不一致，触发合规/索赔风险。SkiillLen 评估要求所有承诺**带因果 + 留替代**——本条异常就是"无因果的硬承诺"。
- **步骤化对策**：
  1. **不承诺**——明确说"该承诺需要证据 + 试用 + 技术负责人确认，本轮不输出"。
  2. **列承诺的前提条件**——证据清单（工况、试用结果、技术评估） + 责任人。
  3. **改用允许话术**——用 13-compliance-and-commitment.md 的"允许话术"模板。
  4. **走 red 升级**——本条异常触发即标 red。
- **允许说什么**：「排放/寿命/替换效果可以作为目标记录，但能否承诺需要结合完整工况 + 样品/报告 + 试用结果 + 技术负责人确认。」
- **禁止说什么**：「我们保证…」「肯定能…」「一定能…」（任何确定性承诺）。
- **升级条件**：**必须升级**——质保/排放/合同/招投标/索赔类承诺 → 技术负责人 + 销售负责人复核。

## 责任归因压力

- **触发信号**：客户问「是不是滤料质量问题」「能不能索赔」「上家掺旧料吗」。
- **机制（为什么会触发）**：责任归因 = 索赔链路起点。**Agent 在证据链未建立前的归因 = 站队**——站客户一边（认质量问题）→ 后续供应商拒赔 → 客户反过来告 Agent 推荐失实；站供应商一边（否认问题）→ 客户失去信任。**两条路都输**。**不站队 = 把责任问题隔离到证据链后**。
- **步骤化对策**：
  1. **不承认不否认**——明确说"在证据链建立前，不做质量归因"。
  2. **列证据需求**——同批是否同工况、是否有检测、是否有工况变化、是否有时序记录。
  3. **走证据链流程**——`references/13-compliance-and-commitment.md` 的"索赔责任"节。
  4. **转人工**——索赔类责任判断必须人工技术确认。
- **允许说什么**：「责任判断需要证据链。请先发 ① 同批滤袋运行记录 ② 工况逐月变化 ③ 失效样品检测报告。」
- **禁止说什么**：「应该是…」「可能是上家…」「肯定不是我们的…」（任何归因）。
- **升级条件**：**必须升级**——索赔/质量责任类 → 技术负责人 + 法务/销售负责人复核。

## 全量替换压力

- **触发信号**：客户/业务员要求「无试用边界就全量替换竞品 / 一次性全部切换」。
- **机制（为什么会触发）**：全量替换 = 客户一次性承担全部风险。**无试用边界 = 不可回退**——一旦替换后新滤袋不达预期（寿命短 / 糊袋 / 排放超标），**没有并行的原方案做对比**，无法定位是替换引发还是新工况引发，**双方都会陷入"是不是对方责任"的循环**。
- **步骤化对策**：
  1. **不直接答应**——明确说"全量替换建议分阶段推进"。
  2. **推试用验证**——走 `references/19-trial-validation-plan.md`，推分仓 POC / 小批量试用。
  3. **设试用边界**——明确试用时长、对比基线、判定标准、回退条件。
  4. **走 red 升级**——本条异常触发即标 red。
- **允许说什么**：「全量替换前建议先做分仓 POC / 小批量试用，时长 X 个月，试用边界为…。」
- **禁止说什么**：「可以一次性全量替换…」（绕过试用边界直接承诺）。
- **升级条件**：客户拒绝任何试用边界 + 仍要求全量替换 → 标 red，转人工。

## 责任归因陷阱

- **触发信号**：客户/业务员直接断言「上家掺旧料 / 你们材质有问题 / 竞品不行」，要求我方站台归因。
- **机制（为什么会触发）**：这是 **framing trap**——把"陈述"包装成"结论"，诱导 Agent 接受预判。**Agent 一旦接受"上家掺旧料"作为前提，后续所有推荐都会偏向"必须换供应商"，与现场真实根因脱钩**（真实根因可能是工况漂移、清灰不足、骨架问题）。
- **步骤化对策**：
  1. **不承认不否认**——明确说"在没有检测数据前，不接受该归因"。
  2. **拆出 framing**——把"上家掺旧料"标记为 framing，不作为事实处理。
  3. **列反问证据**——同批是否同工况、是否有检测、是否有工况变化、是否有时序记录。
  4. **回到我方工况**——所有判断回到"现场实际是什么"，不回到"谁的责任"。
- **允许说什么**：「"上家掺旧料"是陈述还是检测结论？请发同批检测报告 + 工况对比。」
- **禁止说什么**：「确实可能是上家问题…」「上家质量确实不行…」（任何接受 framing 的回应）。
- **升级条件**：客户坚持 framing + 拒绝检测 → 标 red，转人工。

## 保密越界

- **触发信号**：客户带竞品出厂报告 / 检测数据 / PPT 资料 / 价格信息；业务员顺手评价竞品。
- **机制（为什么会触发）**：保密越界 = 商业风险 + 法律风险。**评论竞品 = 留下书面证据**，后续可能用于反垄断诉讼 / 不正当竞争诉讼。**外传客户资料 = 违反保密协议**。**评价越多，证据越多**。
- **步骤化对策**：
  1. **不评论竞品**——明确说"对竞品不做技术/价格/质量评价"。
  2. **资料归档脱敏**——所有竞品资料仅作内部技术参考，不外传、不引用、不作对外承诺依据。
  3. **回到我方工况**——所有判断回到"我方产品在客户工况下的表现"，不与竞品直接对比。
  4. **走保密协议**——涉及客户资料/竞品资料的，标注使用范围与责任人。
- **允许说什么**：「资料收到，仅作内部参考。所有判断基于我方产品在贵司工况下的表现，不与竞品直接对比。」
- **禁止说什么**：「竞品确实不如…」「竞品价格更低…」「竞品有 XX 问题…」（任何竞品评论）。
- **升级条件**：业务员/客户要求书面评价竞品 → 标 red，转销售负责人 + 法务。

## 客户原话丢失

- **触发信号**：微信/会议纪要里业务员转述掺入自己的「是不是材质问题 / 是不是上家不行」等评价；客户原话没被保留。
- **机制（为什么会触发）**：业务员转述 ≠ 客户原话。**评价混入转述 = framing trap + 责任归因前置**——后续 Agent 拿到的"客户陈述"已经包含业务员的价值判断，按此推进 = 接受预判。
- **步骤化对策**：
  1. **强制分离字段**——把 `customer_original_words`（客户原话）与 `salesperson_interpretation`（业务员评价）拆成两个独立字段。
  2. **缺原话时标 context_gap**——客户原话字段为空 = 上下文缺口，不基于此输出。
  3. **补原话**——要求业务员提供客户原话（截图 / 录音 / 邮件）。
  4. **降承诺等级**——原话缺失 = Level ≤ 1。
- **允许说什么**：「请把客户原话与您的评价分开。原话（微信截图 / 邮件）发我后再继续。」
- **禁止说什么**：「既然客户/您觉得是材质问题…」（把转述当事实）。
- **升级条件**：连续 2 轮仍拿不到客户原话 → 标 amber，转人工。

## 异常状态的输出强制结构

> 任何异常触发时，本轮输出**必须**包含以下 5 段。缺一段即视为绕过异常处理。

```markdown
## 异常状态：<类型名>

1. **异常类型**：<从上述 12 类中选 1 个或叠加>
2. **为什么这是异常**：<机制 + 当前陈述与事实的差距>
3. **现在允许说什么**：<具体允许话术，可直接复制给客户>
4. **现在禁止说什么**：<具体禁止话术 / 禁止行为>
5. **下一步需要谁补什么证据**：<责任人 + 必查证据清单 + 时限>
```

## 升级规则

以下异常**必须升级到人**（不止 Agent 处理）：

- 质保、寿命、排放、合同、招投标承诺 → 销售负责人 + 技术负责人
- 客户索赔或质量责任判断 → 技术负责人 + 法务/销售负责人
- 大项目全量替换 → 销售负责人 + 技术负责人 + 管理层（按金额）
- 安环风险（火星烧袋、爆炸、环保处罚、安全事故）→ 技术负责人 + 安环负责人
- 客户资料 / 竞品报告 / 检测数据涉及保密边界 → 销售负责人 + 法务
- framing trap + 客户坚持 + 拒绝证据 → 销售负责人 + 技术负责人
