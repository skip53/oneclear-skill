# 29｜进度显示格式 (Progress Visibility)

本文件定义了每轮响应必须包含的进度可见性信息。业务员需要在每轮看到"我们现在在哪、还差什么、还要多久"。

## 每轮进度条最小元素

| 字段 | 含义 | 示例 |
|---|---|---|
| 轮次 | 当前是第几轮 | 第 2 轮 / 预计 4-6 轮 |
| Level | 当前承诺等级 | Level 2 |
| 案件状态 | state machine 阶段 | discovery / diagnosis / direction / candidate-review / blocked |
| 任务类型 | Round 0 锁定 | B 旧项目故障 |
| 已知数 | 已识别 P0+P1 字段数 | 8 / 12 |
| 未知数 | 仍缺关键字段数 | 4 / 12 |
| 矛盾数 | 矛盾信息条数 | 1 |
| 决策点数 | 本轮 AskUserQuestion 调用题数 | 2 / 4(能答项 + 推进方) |
| 预计剩余轮数 | 估算 | 2-3 轮收敛 |

"决策点数" 字段是 AskUserQuestion 集成后新增的——让业务员在每轮一眼看出"这一轮有几个方向键选择题要答"。完整决策点矩阵见 `references/30-askuserquestion-integration.md`。

## Round 输出固定格式(SKILL.md 主入口也引用)

```markdown
## 第 X 轮 / 预计 N 轮 · 当前 Level L · 决策点 N 个

- 任务类型: ...
- 案件状态: ...
- 本轮新增: ...
- 上一轮结论: ...

### 已知信息 (✅)
- [已知] ...

### 未知关键信息 (❌)
- [未知] ...

### 矛盾信息 (⚠️)
- [矛盾] ...

### 初步风险标签
- ...

### 本轮追问 (3-7 个)
#### 内部技术版
1. ...
#### 客户转述版
1. ...

### 本案承诺红线 (持续更新)
- ...

### 下一步动作
- 负责人: ...
- 推进目标: ...
- 收敛触发条件: ...

[本轮结尾追加 AskUserQuestion 多题批量调用,见 27-interactive-round-protocol.md 的 Round 2+ 节]
```

进度标题里的 "决策点 N 个" 字段让业务员立刻看到本轮需要用方向键回答多少题,避免漏答。AskUserQuestion 调用本身在响应末尾,不在进度条里展开——展开会破坏 markdown 进度条的可读性。

## 进度条的语义规则

### Round 编号

- Round 0 = 任务识别
- Round 1 = 建档与分流
- Round 2+ = 逐轮补全
- 收敛 = 不算 Round,显式标 "Convergence"

### 预计 N 轮

- N 是粗估,不承诺。
- N = 剩余 P1 字段数 / 5(每轮可处理约 5 个 P1)+ 1。
- N 上限 7,超过强制收敛。

### Level 显示

- Level 0 = 无法分流(只 Round 0 或 Round 1 极端情况)
- Level 1 = 可分流,不可诊断
- Level 2 = 可初步诊断,不可选型
- Level 3 = 可给选型方向
- Level 4 = 可给候选方案(需人工确认)
- Level 5 = 可形成技术支持包

### 案件状态

| 状态 | 含义 | 出现条件 |
|---|---|---|
| intake | 首轮建档中 | Round 1 |
| discovery | 缺关键 P0/P1 | Round 2+ |
| diagnosis | 可初步诊断 | Level ≥ 2 之后 |
| direction | 可给方向 | Level ≥ 3 之后 |
| candidate-review | 有候选方案,待人工确认 | Level ≥ 4 之后 |
| support-pack | 可形成技术支持包 | Level ≥ 5 |
| blocked | 异常终止 | 见 Block 触发条件 |

### 已知/未知/矛盾 的视觉规则

- ✅ 已识别且无矛盾 → 已知
- ❌ 关键字段缺失 → 未知(必须 P0/P1 才计)
- ⚠️ 客户说法自相矛盾 → 矛盾(必须双方都有原文)
- 🟡 用户跳过 / 待确认 → 单独标,不算未知也不计入进度

> 不要把"待证据"(缺照片/样品/检测报告)算作矛盾;它属于"已知缺失证据",归入 ❌ 未知,标签写"需证据"。

## 进度变化的可视化规范

### 上升(正常收敛路径)

```text
第 1 轮 / 预计 4-5 轮 · Level 1 · 决策点 1 (案件分流) (3/12 已知, 9 ❌, 0 ⚠️)
第 2 轮 / 预计 3-4 轮 · Level 2 · 决策点 2 (能答项 + 推进方) (7/12 已知, 5 ❌, 0 ⚠️)
第 3 轮 / 预计 1-2 轮 · Level 3 · 决策点 3 (能答项 + 推进方 + 新事实) (10/12 已知, 2 ❌, 1 ⚠️)
Convergence · Level 3 · 决策点 1 (收敛门) (12/12 已知, 0 ❌, 1 ⚠️ 已标注)
```

### 收敛后再补充(Refinement)

```text
Refinement · Level 4 · 决策点 1 (收敛门) (12/12 已知 + 1 补充, 0 ❌, 1 ⚠️)
> 上一轮结论: 给出方向 A
> 本轮新增: 收到现场压差曲线,确认方向 A 可行
> Level 变化: 3 → 4 (补充后达到 Level 4 必要 P1)
```

### 强制收敛(超过 7 轮仍未达标)

```text
Forced Convergence · Level 2 · 决策点 1 (收敛门) (5/12 已知, 7 ❌, 2 ⚠️)
> 强制收敛原因: Round 7 已达上限
> 仍未解决: 7 个 P1 字段缺失
> 后续路径: 转人工或等业务员补充数据后再次进入 Round 2
```

### Blocked

```text
Blocked · Level N/A · 决策点 1 (替代路径)
> 触发条件: 客户要求"保证 X 年寿命"
> 当前已知: 12/12 已知
> 替代路径: AskUserQuestion 提供 2-3 条路径供用户选
> 本案红线: 见 禁止承诺项(本案专列)
```

## 进度信息对业务员的可见性收益

业务员应在 5 秒内能从响应顶部看出:

1. 现在在哪一轮。
2. 离结论还有多远。
3. 这轮要回答什么问题。
4. 哪些可以"先按这个推进"。

不要让业务员读完整段响应才能定位进度——进度条必须出现在响应最前。
