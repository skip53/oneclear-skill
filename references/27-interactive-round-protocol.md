# 27｜交互式多轮协议（Interactive Round Protocol）

本协议定义了 oneclear 在每轮"做什么、不做什么、怎么输出"。SKILL.md 是入口,本文件是每轮的执行细则。

## 状态机总览

```text
Round 0  任务识别         → 1 个菜单问题 + 已识别事实
Round 1  建档与分流       → P0 补齐 + 风险标签 + Level 评估
Round 2  主动追问 P1      → 3-7 个问题 + 进度
Round 3  继续补全         → 3-7 个问题 + 进度
...                           每轮重评 Level
Round N  收敛             → 完整输出包 (Level 满足任务时)
Optional Refine 更新包    → 增量更新 + Level 变化说明
Blocked  异常终止         → 风险红线触发,显式升级
```

## Round 0 — 任务识别 (Task Menu)

### 必做

1. 输出任务菜单(见 `references/28-task-menu.md`)。
2. 如果用户已粘贴内容,先做最小抽取(行业、工艺段、客户原话、痛点关键词),用"已识别到的事实"框展示,再呈现菜单。
3. 如果第一句话里任务非常明确(例如"我先报个价"),可作为推荐默认项,但仍要把完整菜单打出来供用户改。

### 必不做

- 不做任何技术判断。
- 不推荐任何材料。
- 不调用 `check_context_completeness.py` 这种脚本,Round 0 没有足够的 P0 数据。
- 不输出诊断包任何一节。

### 收口条件

- 用户明确选了 A-J 之一 → 进入 Round 1。
- 用户说"按你推荐的来"且已有推荐默认 → 进入 Round 1,沿用推荐。
- 用户在 AskUserQuestion 默认任务确认中选 `确认默认` 或 `调整为: X` → 进入 Round 1。
- 用户在 AskUserQuestion 中选 `其他` 并粘贴原文 → 进入 Round 1(按原文重新判定任务类型)。

### Round 0 — AskUserQuestion 默认任务确认(条件性)

**触发条件**: 用户粘贴了内容 / 第一句话任务明显指向某个 A-J / `classify_case_type()` 高置信度匹配。**不触发**: 纯冷启动 `/oneclear` 无任何内容 / 第一句话完全不明确。

调用细节:

| 字段 | 值 |
|---|---|
| 调用位置 | A-J 菜单 markdown 块**之后**,Round 0 收口之前 |
| 题数 | 1 |
| 类型 | 单选 |
| header chip | `任务确认` |
| 选项 | `确认默认: <推荐任务>` / `调整为: <备选 1>` / `调整为: <备选 2>` / (自动) `其他(可粘原文)` |

选项构造逻辑:
- `确认默认: <推荐任务>` 是默认项,**带 `(推荐)` 标记**,header chip 之后用 `推荐默认任务` 给出更明确提示
- 备选 1/2 从 A-J 中挑与推荐最相邻的两项(例:推荐 D 报价准备 → 备选 A 新项目选型 + F 招投标支持)
- `其他` 选项自动由 AskUserQuestion 加,正好对应 I/J 路径——用户选它后粘贴原文,我们按 I 路径 fast-forward

不要在冷启动场景调 AskUserQuestion——冷启动用户没粘贴任何东西,默认不明显,直接出 A-J 自由文本即可。

### Round 0 输出模板

```markdown
## Round 0 — 任务识别

(若有粘贴内容:)
> **已识别到的事实**: 行业=水泥;工艺段=窑尾袋除尘;痛点关键词=压差高、糊袋…
> **建议任务类型**: B) 旧项目故障
>
> 如果不准,请从下方菜单中重新选择。

你今天需要处理哪个任务? (选一个最贴的,后面可以再调)

A) 新项目选型 — 客户新建项目/新产线,需要确定滤袋
B) 旧项目故障 — 现场滤袋出问题,要找原因/挽救
C) 替换/降本 — 客户已有方案,想换供应商或降本
D) 报价准备 — 客户要报价/询价,需要准备报价资料
E) 试用验证 — 小批量试用、分仓测试、POC 验证
F) 招投标支持 — 招标/投标技术响应、偏差表
G) 竞品对比 — 客户带竞品资料或压价,要应对
H) 复购/补单 — 老客户补单,需要规格/交期确认
I) 我有客户原话/聊天记录,需要整理 → (选此项请直接粘贴原文)
J) 其他 / 不确定

[若默认明显,在 markdown 块之后追加 AskUserQuestion: 任务确认]
```

## Round 1 — 建档与分流 (Intake & Routing)

### 必做

1. 用 `scripts/classify_case_type.py`(或规则)在脑中跑一遍输入,确认任务类型。如果和 Round 0 推荐一致,显式确认;不一致,告诉用户改了什么。
2. 抽取所有可见 P0:`客户原话`、`项目类型`、`行业`。每个抽到的字段写到档案的"已知信息"。
3. 列出本任务类型对应的 P1 列表(见 `references/01-question-routing.md`)。
4. 用 `references/04-risk-tags.md` 打初步风险标签。
5. 用 `scripts/check_context_completeness.py` 的规则在脑中评估 Level。
6. 输出 Round 标准格式(见 SKILL.md 的"Round Output Format")。

### 必不做

- 不抽所有 8 块工况档案,只抽 P0 加上能从输入里直接看出的字段。剩余字段留到 Round 2+。
- 不给具体材料方向。
- 不输出完整诊断包;Round 1 的输出结尾一定是"本轮追问"。

### Round 1 任务对应的 P1 默认追问集

新项目:`工艺段 / 常态温度 / 峰值温度 / 含湿量 / 粉尘性质 / 除尘器类型 / 排放要求 / 目标寿命`

旧项目故障:`工艺段 / 故障表现 / 当前材质 / 使用时长 / 故障位置·比例 / 常态温度 / 含湿量 / 清灰参数 / 压差曲线 / 近期工况变化 / 现场照片·样品·报告`

替换降本:`当前材质 / 当前寿命 / 替换目标 / 当前方案是否稳定`

报价准备:`规格 / 数量 / 工况 / 风险边界 / 客户目标`(这五项是 `references/17-quote-readiness.md` 的最低门槛)

试用验证:`试用范围 / 验收指标 / 对照基线 / 试用周期 / 风险边界`

招投标:`招标技术条款 / 偏差表 / 商务条款 / 评审节点 / 风险边界`

竞品对比:`竞品型号 / 当前基线 / 对比维度 / 客户选择标准 / 替代范围`

复购:`订单规格 / 历史表现 / 数量 / 交期 / 验收标准`

> 任何 P1 中缺 ≥4 项,Round 1 终止在 Level 1;缺 1-3 项进入 Level 2;全部齐进入 Level 3。

### 收口条件

- 用户回答了 ≥1 个 P0/P1 → 进入 Round 2。
- 用户说"我先去看一下现场"等 → 暂停案件,告知"再回来时把现场数据发给我"。
- 用户拒绝提供 → 转 Blocked 状态,显式说明无法推进。

### Round 1 — AskUserQuestion 案件分流确认

在输出工況档案草稿**之前**,插入一次 AskUserQuestion 案件分流确认。**4 选项正好匹配工具上限**,且与 `01-question-routing.md` 的四类分流对齐。

| 字段 | 值 |
|---|---|
| 调用位置 | Round 1 开头,"案件状态"行之前 |
| 题数 | 1 |
| 类型 | 单选 |
| header chip | `案件分流` |
| 选项 | `新项目选型` / `旧项目故障` / `替换·降本` / `信息不足` |

调用结果处理:

| 用户选择 | 处理 |
|---|---|
| `新项目选型` | 走 A 的 P1 默认集;Level 评估按新项目路径 |
| `旧项目故障` | 走 B 的 P1 默认集;Level 评估按故障路径,风险标签加重 |
| `替换·降本` | 走 C 的 P1 默认集;Level 评估按替换路径,商业价值前置 |
| `信息不足` | 走最小补全模式,Round 2 进入强追问,Level 评估为 1 |
| `其他`(自动) | 用户可粘补充,按内容重新分流 |

把选定路径在工況档案里显式标注(例:`路径: 替换·降本`),但不要重复问分流——一次 AskUserQuestion 就够。

## Round 2+ — 逐轮补全 (Progressive Discovery)

### 必做

每轮开始时:

1. 合并用户上一轮提供的新事实到档案。标注 `本轮新增`。
2. 用 `check_context_completeness.py` 规则重评 Level。
3. 用 `risk_tag_rules.py` 规则重打风险标签(若有更新)。
4. 列出仍缺的 P1 字段,按 P0 → P1 → P2 → P3 排序。
5. 选 3-7 个最高优先级,组织成 1 轮问题。
6. 输出 Round 标准格式。

每轮结尾时:

1. 给出下一轮的收敛触发条件(例:Level 达到 3 → 收敛;用户答完本轮问题 → 评估收敛)。
2. 若累计已达 7 轮且 Level 仍 < 任务目标 → 强制收敛,显式列出"仍未解决"。

### 必不做

- 不重复问已答的字段。
- 不在用户已回答的字段上挑刺。
- 不超过 7 问/轮。
- 不在 Round 2-7 输出完整的诊断包,只输出追问 + 进度。

### 多字段批量处理规则

如果用户本轮一次性提供了多个字段(常见:粘贴一个完整对话),不要分多次追问,本轮内:

1. 立刻抽取所有字段,合并到档案。
2. 重评 Level。
3. 直接列出**仍然缺失**的字段,作为下一轮追问。
4. 如果 Level 已达任务目标 → 收敛。

### Round 2+ 输出模板

```markdown
## 第 X 轮 / 预计 N 轮 · 当前 Level L

- 任务类型: ...
- 案件状态: discovery / diagnosis / direction / candidate-review / blocked
- 本轮新增: <1-3 行>
- 上一轮结论: <1-2 行>

### 已知信息 (✅)

- [已知] 行业: 水泥
- [已知] 工艺段: 窑尾袋除尘器
- [已知] 常态温度: 180℃
- ...

### 未知关键信息 (❌)

- [未知] 当前滤料材质 — 决定是否有 baseline
- [未知] 入口粉尘浓度 g/Nm³ — 决定过滤负荷边界
- ...

### 矛盾信息 (⚠️)

- [矛盾] "灰比较干" vs "压差上升快、灰层发硬" — 影响: 单纯干灰不应发硬,可能漏风/局部冷点

### 初步风险标签

- 温度(波动 150-200℃+)
- 氧化(O₂ 8-10%,PPS 类窗口收窄)
- ...

### 本轮追问 (3-7 个)

#### 内部技术版

1. 问题: 当前在用滤袋的材质型号是什么?使用了多久?
   - 优先级: P0
   - 对应字段: 当前材质 / 已使用时长
   - 为什么问: 没有 baseline 无法判断"寿命偏短"的参照系

#### 客户转述版

1. 麻烦帮忙确认下现在在用的滤袋是什么材质型号,已经用了多久,最好能给到实际运行小时数。
2. ...

### 本案承诺红线 (持续更新)

- 不承诺具体寿命年限 — 实际寿命受温度·O₂·含湿·清灰·压缩空气综合影响
- 不承诺具体排放数值 — 排放由滤料·设备·工艺·管理综合决定
- 不下结论"上家滤料质量差" — 缺证据前不做此类指控
- 不锁定单一型号推荐 — Round 2 阶段不应给唯一型号

### 下一步动作

- 负责人: 业务员收集现场数据;技术支持核验并起草档案
- 推进目标: 完成本轮 P0/P1 后进入下一轮,预计 X 轮收敛
- 收敛触发条件: Level ≥ 3 且 P0/P1 完整 / 用户明确说"出方案"
```

### Round 2+ — AskUserQuestion 多题批量(每轮结尾)

每轮 3-7 开放追问**以自由文本 markdown 列出**(开放问答不适合 AskUserQuestion),但**每轮结尾追加一次 AskUserQuestion 多题批量调用**,把本轮的相关决策点一次性问完。

#### 调用位置

`### 下一步动作` 块之后,Round 收尾之前。

#### 调用模板(根据本轮实际决策点填充 2-4 题,所有题共用一次 AskUserQuestion 调用)

| 题序 | 题型 | header chip | 选项来源 | 默认 |
|---|---|---|---|---|
| 1 | multiSelect | `能答项` | 本轮 3-7 追问的标题(去掉标点,压缩到 ≤25 字) | 无 |
| 2 | 单选 | `推进方` | `业务员` / `技术支持` / `客户` | `业务员` |
| 3(可选) | 单选 | `新事实` | `无新事实` / `有新事实` / `客户在确认中` | `无新事实` |
| 4(可选) | 单选 | `收敛门` | `继续追问` / `收敛出包` | `继续追问` |

#### 各轮次的题数选择

| Round | 推荐题数 | 原因 |
|---|---|---|
| Round 2 | 1+2(2 题) | 刚开始追数据,只问"能答哪些"+"谁催",避免给业务员压力 |
| Round 3 | 1+2+3(3 题) | 数据回了一些,加"新事实"判断是否还要再追 |
| Round 4 | 1+2+3(3 题) | 同上,Level 通常升到 3,可试探收敛方向 |
| Round 5 | 1+2+3+4(4 题) | Level 应达 3-4,加"收敛门"试探 |
| Round 6+ | 1+2+4(3 题) | 已接近强制收敛门,收敛门必问 |

> Level 未达任务目标时**不要出 4 收敛门**——硬出收敛门会让用户选 "继续追问",反而打断节奏。

#### "能答项" 选项构造规则

- 每条选项格式: `Q<n>: <标题压缩版>`
- 标题去掉标点(冒号、问号、顿号、破折号、引号)
- 压缩到 ≤25 字(超出截断 + "...")
- AskUserQuestion 单题最多 4 选项——若本轮追问 > 4,只暴露最高优先级的 4 条进 AskUserQuestion,其余留在 markdown 列表里
- 选项要可读——"Q1: 短时高温实测持续时间" 比 "Q1: 短时高温 ≥200℃ 实测细节" 好

#### 调用结果处理

提交后,AskUserQuestion 返回的 answers 合并到下一轮:

| AskUserQuestion 题 | 下一轮的处理 |
|---|---|
| `能答项` 多选 | 选中的项目对应的字段进入下一轮的 `### 已知信息 (✅)`,未选的字段保留在 `### 未知关键信息 (❌)` |
| `推进方` 单选 | 下一轮 `### 下一步动作 - 负责人` 自动填充该选项 |
| `新事实` 单选 | 选 `有新事实` → 下一轮先合并新事实再追问;选 `客户在确认中` → 保留案件为 `intake` 状态 |
| `收敛门` 单选 | 选 `收敛出包` → 立即进入 Convergence 输出包(若在本节已问,Convergence 节不再重复问) |

## Convergence — 输出包

### 触发条件(满足任一)

1. Level ≥ 任务目标 Level(见 `references/28-task-menu.md` 任务-目标 Level 映射表)。
2. 用户显式说"出方案/汇总/够了/下一步/可以了"。
3. Round ≥ 7,强制收敛。
4. 进入 Blocked 状态。

### Convergence — AskUserQuestion 收敛门(最后一英里)

在判定收敛触发条件成立后、生成输出包之前,插入一次 AskUserQuestion 单题单选,作为"最后一英里"确认——避免在用户没明确点头的情况下直接出包。

| 字段 | 值 |
|---|---|
| 调用位置 | Convergence 节开头,"Output the formal pack"之前 |
| 题数 | 1 |
| 类型 | 单选 |
| header chip | `收敛门` |
| 选项 | `收敛出包` / `继续追问` / (自动) `其他(自由补充)` |

**短路规则**: 如果本轮的"收敛门"已经通过 Round 2+ AskUserQuestion 多题批量问过且用户选了 `收敛出包`,本节**不再重复问**,直接进输出包。

调用结果处理:

| 用户选择 | 处理 |
|---|---|
| `收敛出包` | 输出完整 4 节包(Level ≥ 4 附加售前推进) |
| `继续追问` | 回到 Round N+1,把 `收敛门` 标记为"已拒绝收敛,继续 Round N+1" |
| `其他`(自动) | 用户可粘补充事实 → 进入 Refinement 增量更新模式 |

### 必做

按 `references/09-output-templates.md` 完整输出 4 节:

1. **工况档案草稿** (8 块 schema + 案件状态 + 上一轮结论 + 本轮新增 + Level 是否变化)
2. **追问清单** (内部技术版 + 客户转述版,即使是收敛轮也要列出尚未解决的 P1)
3. **初步诊断摘要** (含下一步行动清单)
4. **禁止承诺项 (本案专列)** — 永远在最末,作为收口节

Level ≥ 4 时附加:
5. **售前推进判断** (决策链 / 报价 readiness / 能力匹配 / 试用边界 / 竞品应对 / 商业价值 / 内部升级 / 人工确认 / 下一步行动)

### 必不做

- 不在 Level 2/3 输出"售前推进判断"(除非用户明确要求)。
- 不在禁止承诺项节里混入正向推荐。
- 不在收敛输出里再次列出追问清单的完整内容,只保留"尚未解决"清单作为风险。

### 收敛输出校验

收敛前自动调用 `scripts/validate_output_pack.py` 的规则做内部检查:

- 4 节齐全。
- 无"保证寿命/排放/质保/全量替代"等越界承诺。
- Level ≤ 2 时无具体材料型号推荐。
- Level 4+ 必须有候选方向 / 证据要求 / 试用边界 / 人工确认。
- Level 5 必须有客户需求摘要 / 工况档案 / 真实需求判断 / 风险假设 / 候选方向 / 试用计划 / 客户话术 / 禁止承诺项 / 下一步行动 / 人工确认状态。

如果校验失败,本轮回到 Round N+1 而不是出包。

## Refinement — 增量更新包

### 触发

- 用户在收敛后又补充了数据。
- 用户反馈"上轮的方向不对,重新看"。
- 内部技术负责人调整了某个判断。

### 必做

1. 输出"上一轮结论 + 本轮新增 + Level 变化 + 更新后的档案 + 仍未解决的问题"。
2. 如果只是细节修正,不需要重新打满 4 节,只输出 diff。
3. 如果是方向性调整,按新事实重新评估 Level,如果退回,转入 Round N+1 重新收敛。

## Blocked — 异常终止

### 触发

- 客户/业务员明确要求承诺被禁止事项(保证寿命、保证排放、质保期限、全量替代必成)。
- 责任归因压力(索赔、质量纠纷)且无证据。
- 信息矛盾持续无法解决(客户说"稳定运行"且"频繁破袋"且拒绝澄清)。
- 商业/合规边界突破(招投标盖技术章、竞品资料保密)。

### 必做

1. 显式标注 Blocked,给出触发条件。
2. 列出当前已知 + 缺失 + 矛盾。
3. 列出当前不能承诺的清单(直接引用本案专列禁止承诺项)。
4. 给出 2-3 条替代路径(转人工/转测试/转基线核对)。

### 必不做

- 不要为了"推进成交"绕过 Blocked。
- 不要在 Blocked 后再问技术问题。
- 不要给"先报个价吧"之类的让步。

### Blocked — AskUserQuestion 替代路径

Blocked 后**不再问技术问题**,而是把 2-3 条替代路径装进 AskUserQuestion,用户点方向键选。

| 字段 | 值 |
|---|---|
| 调用位置 | Blocked 状态显示之后,本节结尾 |
| 题数 | 1 |
| 类型 | 单选 |
| header chip | `替代路径` |
| 选项 | 2-3 条具体路径(见下表) |
| `其他`(自动) | 用户可自定义路径 |

按 Blocked 触发条件匹配替代路径:

| Blocked 触发 | 替代路径选项 |
|---|---|
| 客户要求禁止承诺项 | `转人工技术负责人复核` / `拒绝该承诺并提供技术分析` / `转为试用验证(分仓 POC)` |
| 责任归因压力无证据 | `转人工技术负责人复核` / `等客户提供证据后重启` / `保持中立,出具第三方检测委托建议` |
| 信息矛盾无法解决 | `转基线核对(等客户澄清矛盾)` / `转人工技术负责人现场勘查` / `降为监控,不推进方案` |
| 商业/合规边界突破 | `转内部合规负责人` / `转人工技术负责人复核条款` / `拒绝盖章并解释边界` |

Blocked 的完整响应结构:

```markdown
## Blocked · Level N/A

> 触发条件: <具体触发>
> 当前已知: <关键字段简列>
> 缺失/矛盾: <不能继续推进的关键缺口>
> 本案红线: <直接引用本案专列禁止承诺项>

[替代路径 AskUserQuestion 调用]
```

用户选完路径后,转对应的下一步动作(转人工则案件状态保持 blocked 但加 `等人工复核` 标签;转测试则转 E 试用验证路径;等等)。
