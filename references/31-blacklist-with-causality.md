# 31｜高风险动作黑名单（带因果）

> 来源：本文件是把 `SKILL.md` 末尾 17 条 Forbidden Behavior 按 **"绝不要 X，因为 Y（机制 / 损失），如果必须做则改用 Z（替代动作）"** 三段式改写后的因果黑名单。SKILL.md 末尾现在只保留一句话指向本文件，**具体禁止动作 + 因果 + 替代全部在本文件**。
>
> 使用方式：每条黑名单都按 `#N` 编号，违反时必须**显式承认**（"已知违反 #N，但因 <具体业务约束> 仍执行"），**不允许静默绕开**。
>
> 与 SkiillLen.md 的对应：D3（高风险动作黑名单）的标准是"绝不要做 X，因为 Y 会导致 Z"——本文件 17 条全部满足该结构。

## 黑名单总表

| # | 绝不要做（X） | 机制 / 损失（Y） | 替代动作（Z） |
|---|---|---|---|
| 1 | 绝不要在 Round 0 一键跑 9 步 pipeline 出诊断包 | 因为 Round 0 还没有 P0/P1 工况数据，跑全流程会基于"业务员转述"虚构"客户工况"→ 方案与现场失真 | 把转录当 fast-forwarded Round 1，提取可见 P0 → 落到 Round 2 的最小问题集 |
| 2 | 绝不要因为输入"看起来明显"就跳过任务菜单 | 因为任务类型决定 P0/P1 字段、Level 评估、禁止承诺清单——跳过 = 后续判断无锚点 | 总是用任务菜单（A-J）确认；若确实明显，可作为"推荐默认"+ 其他选项 |
| 3 | 绝不要在工況不足时推荐具体滤料型号 | 因为材质推荐与温度/含湿/粉尘/清灰强耦合，无数据推荐 = 拿客户现场做实验 | 改给"选型方向 + 必查的 P0/P1 清单"，等 Level 3+ 再给具体型号 |
| 4 | 绝不要在无诊断时把客户的表层需求当真实问题 | 因为表层需求 ≠ 真实工况问题 ≠ 商业目标，三者冲突时按表层推进 = 客户拿到不解决现场问题的方案 | 走真实需求验证（10-true-demand-validation.md），拆开三层后再定方向 |
| 5 | 绝不要在无证据时把糊袋/破袋/压差高/排放超标直接归因为滤料质量问题 | 因为这些是结果症状，根因可能在清灰/风速/含湿/骨架/破袋——单点归因 = 错失真正修复点 + 客户反向索赔风险 | 走 06-failure-diagnosis-playbook.md 的"机制 + 对策"流程 |
| 6 | 绝不要回声客户/业务员的预判 framing（"上家掺旧料/竞品不行/便宜差不多") | 因为 framing 是陈述包装成结论——按结论推进 = 接受预判、与现场根因脱钩 | 拆出 framing 标记为 `[framing]`，要求证据链；走 11 异常节"责任归因陷阱"流程 |
| 7 | 绝不要把"客户说便宜/寿命差不多/全量替换"当作自证的选择路径 | 因为这些是商业压力信号，不是技术判断——按此推进 = 绕过试用边界 | 把这些信号当作"要求 baseline + trial boundary"的触发器（04 替换降本风险） |
| 8 | 绝不要把"客户说稳定"与"客户说频繁破袋/要求全量替换"作为一致图景推进 | 因为这两组陈述在同一时间窗下几乎必有一方失实——按任一方推进 = 拿矛盾当事实 | 标 `contradictory_facts`，拆时间窗/拆口径；不解决矛盾不进入 Level 4+ |
| 9 | 绝不要承诺寿命、排放、替换效果、保证降本 | 因为这些都需要工况 + 试用 + 技术负责人确认，Agent 无证据下承诺 = 触发合规/索赔风险 | 用 13-compliance-and-commitment.md 的"允许话术"模板（"目前可作为目标记录，但能否承诺需..."） |
| 10 | 绝不要凭空提供/估算/引用具体单价、价格区间或总成本 | 因为价格与工艺/材质/订单量/付款条件强耦合，无源数据 = 失真报价 → 客户按错价签约后违约 | 价格信息**只能**来自客户直接报价、供应商盖章报价单、或带源链接的公开价目表；预算上限仅用作计算上限，不作隐含市场价 |
| 11 | 绝不要在一轮内给业务员甩一长串检查清单 | 因为长清单 = 业务员读不完、答不下、节奏断——多轮交互失效 | 每轮 3-7 个 P0/P1 追问（最高优先级），其余分轮补 |
| 12 | 绝不要跨轮重复问同一个 P0 问题 | 因为重复问 = 业务员感觉 Agent 没记忆 + 失去信任 | 工況档案里维护"已问过"标记；未答标 `[用户跳过,需人工确认]` 后移到下一轮 |
| 13 | 绝不要超过 7 轮还没收敛 | 因为超过 7 轮 = 节奏失控 + 客户/业务员疲劳 + 证据仍不齐 = 注定出不了合格包 | 第 7 轮末尾强制收敛门（AskUserQuestion "收敛门"），明确"仍缺什么"披露 |
| 14 | 绝不要忽略客户原话、只靠业务员转述 | 因为业务员转述会压缩、改写、添加评价——按转述推进 = 把业务员判断当客户工况 | 强制分离 `customer_original_words` 与 `salesperson_interpretation`；缺原话 = 标 `context_gap` |
| 15 | 绝不要在关键风险未披露时推进报价或最终方案 | 因为未披露风险 = 报价/方案失真 → 现场不达预期 → 客户反索赔 | 走 12 风险升级协议，red 状态下不推进报价；报价前必过 17-quote-readiness.md |
| 16 | 绝不要把客户表达的需求当作"真实"，当它与现场事实/商业阶段/证据冲突时 | 因为客户表达 = 主观陈述，与现场/证据冲突 = 表达失真 | 走 11 异常节"信息矛盾"流程；不调和矛盾不进入 Level 4+ |
| 17 | 绝不要忽略合规 / 承诺边界（排放、安环、质保、合同、招投标、客户资料、竞品资料） | 因为这些边界有法律/商业责任——忽略 = 触发合规处罚 + 客户/竞品双向索赔 | 走 13-compliance-and-commitment.md；盖章/承诺类一律技术 + 销售 + 法务三签 |

## 跨条目通用检查清单

> 任何条目触发前先过这 4 条。

1. **是否 framing**？客户/业务员已锁定结论 → 不接受预判，回到证据。
2. **是否触达合规边界**？涉及寿命/排放/替换/索赔/合同/招投标 → 走 13，走 12 升级。
3. **是否触达商业风险**？低价替代/全量替换/降价 30% → 走 04 替换降本风险 + 19-trial-validation。
4. **是否触达责任归因**？客户/业务员要求站台归因 → 走 11 责任归因陷阱。

## 违反时的强制承认格式

> 违反某条黑名单时，本轮输出**必须**包含以下 4 段：

```markdown
## 黑名单违反声明

- **违反条目**：#<N>
- **违反原因（具体业务约束）**：<为什么必须做>
- **风险敞口**：<会造成什么损失，谁承担>
- **补偿动作**：<如何把风险敞口压到最小>
```

不允许静默绕开——任何条目违反都必须有显式承认 + 补偿动作。
