# 30｜AskUserQuestion 集成模式 (Decision Points Integration)

本文件集中说明 oneclear 在哪些决策点使用 Claude Code 的 `AskUserQuestion` 工具,以及为什么其他位置不使用。它是 27 / 28 / 29 三个 references 中所有 AskUserQuestion 调用的设计依据。

## 1. 工具约束(必须遵守)

来自 Claude Code 系统提示的硬约束:

| 约束 | 数值 | 含义 |
|---|---|---|
| 单次调用最多题数 | **4** | 一次 `AskUserQuestion` 调用最多装 4 个问题 |
| 每题最多选项 | **4** | 每个问题最多 4 个选项 + 工具自动加的 `其他` |
| 单题类型 | 单选 / 多选 | 通过 `multiSelect` 字段切换 |
| header chip 长度 | **≤12 字符** | chip 显示在题目旁,超出会被截断 |
| 自动 `其他` 选项 | 必有 | 用户选 `其他` 后可自由输入文本 |
| Submit Answer | 自动 | 用户填完所有题目后点一次提交全部答案 |

这些约束决定了下文每个决策点的具体设计——为什么是这个题数、为什么是这些选项。

## 2. 决策点矩阵

| 决策点 | 用 AskUserQuestion? | 题数 | 类型 | header chip | 选项数 | 出现轮次 |
|---|---|---|---|---|---|---|
| Round 0 A-J 任务菜单 | ❌(超 4 选项) | - | - | - | 10 | Round 0 |
| Round 0 默认任务确认 | ✅ | 1 | 单选 | `任务确认` | 2-3 | Round 0(条件性) |
| Round 1 案件分流 | ✅ | 1 | 单选 | `案件分流` | 4 | Round 1 |
| 每轮"本轮能答哪些题" | ✅ | 1 | 多选 | `能答项` | 3-7 | Round 2+ |
| 每轮"下一步谁推进" | ✅ | 1 | 单选 | `推进方` | 3 | Round 2+ |
| 每轮"是否有新事实" | ✅(可选) | 1 | 单选 | `新事实` | 3 | Round 3+ |
| 收敛门 | ✅ | 1 | 单选 | `收敛门` | 2 | Round 5+/Convergence |
| 替代路径(Blocked) | ✅ | 1 | 单选 | `替代路径` | 2-3 | Blocked |
| 3-7 开放追问 | ❌(开放问答) | - | - | - | - | Round 2+ |
| 工況档案 / 风险标签 / 输出包 | ❌(纯展示) | - | - | - | - | 所有轮 |

## 3. 什么时候用 AskUserQuestion

满足以下全部条件才用:

1. **选项 ≤ 4 个**——A-J 菜单(10 选项)因此不能用
2. **选项互斥(单选)或可选多个(多选)**——开放式追问因此不能用
3. **用户答完即可继续推进**——纯展示块(工況档案)因此不能用
4. **方向键选择能改善体验**——例如"案件分流" 4 个选项并列,方向键切换比打字选更直接

## 4. 什么时候不用 AskUserQuestion

| 决策点 | 原因 |
|---|---|
| A-J 任务菜单 | 10 选项超 4 上限;且所有选项并列展示更直观 |
| 3-7 开放追问 | 用户要回答的是"短时高温持续多久" 这种开放问题,不是选择题 |
| 工況档案 / 风险标签 | 纯展示,无决策 |
| 输出包 | 已经收敛,无更多决策 |

## 5. 多题批处理模式(关键设计)

**Round 2+ 每轮结尾把 2-4 个相关决策点合并到一次 AskUserQuestion 调用**,用户点一次 Submit Answer 提交所有答案。

### 各轮次的题数选择

| Round | 推荐题数 | 原因 |
|---|---|---|
| Round 2 | 1+2(2 题) | 刚开始追数据,只问"能答哪些"+"谁催" |
| Round 3 | 1+2+3(3 题) | 数据回了一些,加"新事实"判断是否还要再追 |
| Round 4 | 1+2+3(3 题) | 同上,Level 通常升到 3 |
| Round 5+ | 1+2+3+4(4 题) | Level 应达 3-4,加"收敛门"试探 |
| Round 6+ | 1+2+4(3 题) | 接近强制收敛门,收敛门必问 |

> **不要每轮都堆满 4 题**——硬出"收敛门"会让用户在 Level 没达标时选"继续追问",反而打断节奏。

### 完整 4 题批量调用示例

```python
AskUserQuestion(
  questions=[
    {
      "question": "本轮你能回答哪些追问?",
      "multiSelect": True,
      "header": "能答项",
      "options": [
        {"label": "Q1: 短时高温实测持续时间"},
        {"label": "Q2: 漏风率"},
        {"label": "Q3: 入口粉尘浓度"},
        {"label": "Q4: 过滤风速"}
      ]
    },
    {
      "question": "下一步谁负责推进?",
      "header": "推进方",
      "options": [
        {"label": "业务员", "description": "客户对接"},
        {"label": "技术支持", "description": "技术判定"},
        {"label": "客户", "description": "直接催客户"}
      ]
    },
    {
      "question": "本轮是否有新事实补充?",
      "header": "新事实",
      "options": [
        {"label": "无新事实"},
        {"label": "有新事实"},
        {"label": "客户在确认中"}
      ]
    },
    {
      "question": "本轮是否准备好收敛?",
      "header": "收敛门",
      "options": [
        {"label": "继续追问"},
        {"label": "收敛出包"}
      ]
    }
  ]
)
```

## 6. "其他" 选项使用约定

`AskUserQuestion` 自动在每题加 `其他` 选项,允许用户输入自定义答案。这正好支持几种场景:

| 决策点 | 用户选"其他" 后 |
|---|---|
| Round 0 任务确认 | 粘客户原文 → 触发 I 选项 fast-forward 路径 |
| Round 1 案件分流 | 粘补充背景 → 按内容重新分流 |
| Round 2+ 能答项 | 自定义能答项编号(例如"Q5 和 Q6") |
| Round 2+ 推进方 | 自定义负责人(例如"区域经理") |
| 收敛门 | 粘补充事实 → 进入 Refinement 增量更新模式 |
| 替代路径 | 自定义路径 |

**不要禁用 "其他" 选项**——它是降级兼容和补充信息的关键入口。

## 7. header chip 命名约定

每个 chip 必须 ≤12 字符,与业务语义对齐:

| chip | 字数 | 用途 |
|---|---|---|
| `任务确认` | 4 | Round 0 默认确认 |
| `案件分流` | 4 | Round 1 4 选项分流 |
| `能答项` | 3 | Round 2+ 多选 |
| `推进方` | 3 | Round 2+ 单选 |
| `新事实` | 3 | Round 3+ 单选 |
| `收敛门` | 3 | Round 5+ / Convergence 单选 |
| `替代路径` | 4 | Blocked 单选 |

> 留 1-2 字符冗余以防用户本地化扩展(例如繁体切换时字符数变化)。

## 8. 与现有 markdown 输出的衔接规则

1. **每轮最后才出 AskUserQuestion**——不要打断已知/未知/矛盾/风险/追问/红线的展示
2. **调用结果合并到下一轮**——不要把 AskUserQuestion 结果单独输出
3. **"能答项" 选中的字段** → 下一轮 `### 已知信息 (✅)`
4. **"推进方" 选项** → 下一轮 `### 下一步动作 - 负责人`
5. **"新事实" 选 `有新事实`** → 下一轮标题加 `(本轮含新事实)`
6. **"收敛门" 选 `收敛出包`** → 立即进入 Convergence 输出包(若已在 Round 2+ 答过,Convergence 节不再重复问)

## 9. 兼容性保障(不破坏现有流程)

| 现有能力 | 集成后状态 |
|---|---|
| `classify_case_type.py` / `assess_context_level()` 等脚本 | 不动,逻辑全部保留 |
| 工況档案 / 风险标签 / 输出包 markdown 块 | 原样保留,不变 |
| 开放追问(3-7 个)的内部技术版 + 客户转述版 | 原样保留,不变 |
| 冷启动 `/oneclear` 无任何内容路径 | 不调 AskUserQuestion,只出 A-J 自由文本 |
| 用户跳过 AskUserQuestion 直接打字 | 工況档案照常更新,流程不卡 |
| Fast-forward 长转录路径 | 不变,只是末尾追加默认确认 AskUserQuestion |
| 强制收敛(Round ≥ 7) | 不变,只是最后用 AskUserQuestion 问"收敛门" |
| 7 轮上限 | 不变,AskUserQuestion 不延后这个上限 |

## 10. 验证场景(集成完成后跑这 8 条)

1. **冷启动无输入**: `/oneclear` 不带任何参数 → 只显示 A-J 自由文本,无 AskUserQuestion
2. **冷启动 + 默认明显**: 粘贴明显是"报价准备"的转录 → A-J + 一次 AskUserQuestion(单题"任务确认",默认标"报价准备")
3. **Round 0 → Round 1**: 用户选 "D 报价准备" → Round 1 开头应有 AskUserQuestion 案件分流(4 选项)
4. **Round 2+ 多题批量**: 在任一轮输出后,出现一次 AskUserQuestion,包含 2-4 个相关决策点
5. **方向键 + Other**: 验证 AskUserQuestion UI 能用方向键切换、能选"其他"自由输入、能多选、能 Submit Answer 批量提交
6. **Other 路径**: 用户在 Round 0 "任务确认" 选"其他"粘原文 → 触发 I fast-forward 路径
7. **收敛门**: Level 达标的轮末尾出现 AskUserQuestion "收敛门";选"收敛出包"才进输出包
8. **降级兼容**: 用户跳过 AskUserQuestion 直接打字回答 → 工況档案照常更新,流程不卡

每条用例跑通即视为集成完成。

## 11. 不要做的事

- ❌ 不要把 A-J 菜单硬塞进 AskUserQuestion(超 4 选项)
- ❌ 不要把开放追问装进 AskUserQuestion(开放问答不适合)
- ❌ 不要在工況档案/风险标签/输出包里夹 AskUserQuestion(纯展示块)
- ❌ 不要每轮都堆满 4 题,Level 未达时硬出"收敛门"
- ❌ 不要禁用"其他"选项(它是降级兼容和补充信息的入口)
- ❌ 不要在 AskUserQuestion 调用前后输出大段说明文字(打断方向键体验)
- ❌ 不要在 AskUserQuestion 调用位置之外的地方要求"选择题回答"——保留开放追问的自由文本
